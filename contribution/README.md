## Prerequisites

Required to develop and run the agent:

- Go 1.24.0+
- Linux kernel 5.4+ with eBPF support (6.x recommended for the full LSM / fentry surface)
- BTF at `/sys/kernel/btf/vmlinux` (most distro kernels ship it)
- Clang 14+
- LLVM 14+
- Containerd (the agent reads the containerd CRI socket for pod / container enrichment)

Optional, enabled automatically when present:

- BPF-LSM (`lsm/` programs). When absent the agent falls back to `kprobe/security_*` with no user action.

Additionally required only for container image builds or the docker-compose single-host mode:

- **Docker** ã used by `make build-image` at the repo root

Kubernetes is only required when you want to run the DaemonSet or
exercise the CRDs end-to-end (`HookSubscription`, `BaselinePolicy`,
`BehaviorContract`, `NodeCapabilities`). Unit tests and
contract adapters run without a cluster.

## Development Environment Setup

### Install Go

```bash
cd contribution/golang
./install-golang.sh
```

### Install BPF Development Tools

```bash
cd contribution/bpf
./install-deps.sh
```

The compiled `.bpf.o` files under `internal/sensor/bpf/` are
tracked in git so plain `go build` works on hosts without clang. Only
rebuild after editing a `.bpf.c` source.

### Install Containerd

Required ã KloudLens reads the containerd CRI socket to enrich raw
syscall events with pod / container / image metadata.

```bash
cd contribution/containerd
./install-containerd.sh
```

### Install Kubernetes Tools

Required only to run the DaemonSet and exercise CRDs end-to-end.

```bash
cd contribution/k8s
./install-kubeadm.sh
./initialize-kubeadm.sh
CNI=flannel ./deploy-cni.sh
```

### Install Docker (Optional)

Only required to build the KloudLens container image
(`make build-image` at the repo root) or to run the docker-compose
single-host deployment under `deployments/docker/`. KloudLens does
**not** talk to the Docker daemon at runtime ã it reads containerd
directly.

```bash
cd contribution/docker
./install-docker.sh
```

## Build and Run

The Go module lives at the repo root. The Makefile drives BPF builds,
formatting/lint/sec gates, unit tests, and container-image builds.

### Build the Agent

```bash
make # bpf + gofmt + golangci-lint + gosec + build
 # ã bin/kloudlens
make bpf # (re)compile the eBPF object under bpf/; only needed
 # after editing .bpf.c sources
make test # go test ./... -v -count=1
make build-image # docker build boanlab/kloudlens:<TAG>
make run # sudo -E bin/kloudlens
```

`make` (the default `build` target) chains gofmt, golangci-lint, and
gosec before compiling, so a plain `make` is enough for local
pre-commit verification. Run `make test` separately.

Live eBPF smoke tests (require root + a supported kernel):

```bash
KLOUDLENS_LIVE_SENSOR=1 go test ./internal/sensor/...
```

### Run the Agent

On Kubernetes:

```bash
kubectl apply -f deployments/crds/
kubectl apply -f deployments/manifests/daemonset.yaml
```

On a host (requires root or `CAP_BPF` + `CAP_PERFMON`):

```bash
sudo ./bin/kloudlens \
 --enrich=cri \
 --cri-endpoint=unix:///run/containerd/containerd.sock \
 --wal-dir=/var/lib/kloudlens/wal \
 --subscribe-addr=127.0.0.1:8181 \
 --admin-addr=127.0.0.1:8180 \
 --metrics-addr=127.0.0.1:9090
```

Or via docker-compose for a single-host dev loop:

```bash
docker compose -f deployments/docker/docker-compose.yaml up
```

### Regenerate gRPC stubs (only when editing `.proto`)

```bash
make proto # protobuf/event.proto ã protobuf/event.pb.go + event_grpc.pb.go
```

The generated `.pb.go` files are tracked in git, so ordinary builds
do not need `protoc` installed.

### Build Release Artifacts

```bash
cd deployments && make flat-manifest # dist/kloudlens-<version>.yaml (CRDs + manifests)
```

## Codebase Map

| Area | Location | Notes |
|---|---|---|
| eBPF tracer sources | `bpf/` | Compiled with `make bpf`; output embedded into the binary via `go:embed` |
| eBPF object (tracked) | `internal/sensor/bpf/` | `kloudlens_<arch>_bpfel.o` ã regenerated by `make bpf`; committed so plain `go build` works without clang |
| Sensor / loader | `internal/sensor/` | Attaches tracepoints, kprobes, fentry, cgroup_skb; owns the eight category ring buffers (crit + bulk_file/net/proc + bulk_file_meta + dns/proc_lc/sock_lc) |
| Wire decode | `internal/bpf2frame/` | Ringbuf record ã typed `wire.Frame` (handles ENTER/EXIT pairing at the wire level) |
| Frame ã intent bridge | `internal/frame2intent/` | `wire.Frame` ã `types.SyscallEvent` with NS/CRI metadata stamped |
| Enricher | `pkg/enricher/` | /proc NS scanner + containerd CRI snapshot + cgroup_id ã pod attribution |
| Intent aggregation | `internal/intent/` | FileOpen / NetworkExchange / ProcessStart / ãÎ coalescing |
| Causal session graph | `internal/graph/` | Fork/exec/file-touch/IPC edges, session queries, optional bbolt persistence |
| Correlation detector | `internal/correlation/` | DNS-answer ã connect cross-references, writeãexec patterns |
| Behavioural baseline | `pkg/baseline/` | Learner + detector + `Profile` snapshot |
| BehaviorContract IR + adapters + gap | `pkg/contract/` | Canonical YAML IR, the seccomp/AppArmor/KubeArmor/Cilium/Kyverno/OPA/NetworkPolicy/PodSecurity adapters, and Profile ã policy gap analysis (`gap.go`) |
| HookSubscription / preset policy | `internal/policy/`, `pkg/policyspec/` | Built-in presets + admission validator |
| WAL | `internal/wal/` | Durable event log with cursor resume |
| Subscribe / Admin gRPC server | `internal/exporter/` | `EventService` (subscribe) + per-stream fan-out + IntentExporter client |
| Adaptive downgrade controller | `internal/downgrade/` | Reacts to ringbuf pressure by sampling the bulk_* rings |
| gRPC `.proto` + generated stubs | `protobuf/` | `EventService` + `AdminService` + `IntentExporter` ã separate Go module. `.pb.go` files are tracked; regenerate with `make proto` from the repo root (delegates to `protobuf/Makefile`). |
| Agent entrypoint | `cmd/kloudlens/` | Node daemon main + Pipeline composition |
| Aggregator (separate repo) | https://github.com/boanlab/kloudlens-aggregator | Multi-node fan-in + re-export |
| Deployments | `deployments/` | `crds/`, `manifests/`, `docker/` |

Anything under `pkg/**` is part of the public surface the
`kloudlens-cli` repo consumes as a tagged Go-module dependency.
Anything under `internal/**` is **not** part of that surface.
If you're about to move a type from `internal/` to `pkg/`, check first
whether the CLI actually needs it ã exporting is a one-way decision
that locks compatibility.

**Regenerating the BPF object** (after editing `.bpf.c` files):

```bash
make bpf # runs clang on bpf/kloudlens.bpf.c; writes the .o into
 # internal/sensor/bpf/
# commit the updated .o alongside the .bpf.c change so plain `go build`
# keeps working on hosts without clang
```

## Code Contribution Process

### 1. Select an Issue

Browse open issues at [GitHub Issues](https://github.com/boanlab/KloudLens/issues).

Label reference:

| Label | Description |
|---|---|
| `good-first-issue` | Suitable for first-time contributors |
| `help-wanted` | Tasks open to external contributors |
| `bug` | Confirmed bug reports |
| `enhancement` | New feature requests |
| `needs-cli-change` | Cannot ship until a matching PR merges in kloudlens-cli |

Comment on the issue you intend to work on before starting.

### 2. Fork and Create a Branch

```bash
# Fork the repository on GitHub, then clone your fork
git clone https://github.com/your-username/KloudLens.git
cd KloudLens

# Add the upstream remote
git remote add upstream https://github.com/boanlab/KloudLens.git

# Create a feature or fix branch
git checkout -b feature/your-feature-name
git checkout -b fix/your-bug-fix-name
```

### 3. Implement and Test

Follow existing code style. Run the agent checks and tests inside each
module you touched ã there is no repo-root Makefile:

```bash
make # chains bpf + gofmt + golangci-lint + gosec + build
make test
```

A plain `make` at the repo root is the same lint/sec chain the agent
CI jobs run individually (`make gofmt`, `make golangci-lint`, `make
gosec`), so a green local `make` + `make test` is the best signal the
PR will pass CI.

### 4. Submit a Pull Request

```bash
git add <files>
git commit -m "feat(sensor): brief description of change"
git push origin feature/your-feature-name
```

Open a pull request on GitHub against the `main` branch. Include the related issue number in the description.

If your change touches the gRPC surface, a Contract IR field, or any
other API the `klctl` CLI depends on, open the matching `kloudlens-cli`
PR at the same time and cross-link them. Label this PR
`needs-cli-change` if the CLI change must land first to keep `main`
green.

## Commit Message Convention

```
<type>(<scope>): <subject>

<body>

<footer>
```

Types:

| Type | Description |
|---|---|
| `feat` | New feature |
| `fix` | Bug fix |
| `docs` | Documentation changes |
| `style` | Code style or formatting changes |
| `refactor` | Refactoring without behavior change |
| `test` | Adding or updating tests |
| `chore` | Build process or tooling changes |

Example:

```
feat(sensor): pair connect enter and exit into NetworkDial intent

- Add enter/exit correlation in internal/bpf2frame
- Emit NetworkDial through internal/intent with dst/port/proto
- Cover the happy path and partial-drop scenarios in unit tests

Closes #123
```

## Issue and Pull Request Guidelines

### Bug Report Template

See [`.github/ISSUE_TEMPLATE/bug_report.md`](../.github/ISSUE_TEMPLATE/bug_report.md).

### Feature Request Template

See [`.github/ISSUE_TEMPLATE/feature_request.md`](../.github/ISSUE_TEMPLATE/feature_request.md).

### Pull Request Requirements

- PR title must be concise and descriptive
- Link all related issues
- Include a summary of changes and the reasoning
- All linter checks and tests must pass (`make` + `make test` from the repo root)
- At least two approvals are required before merging
- If the PR needs a matching `kloudlens-cli` change, cross-link it
